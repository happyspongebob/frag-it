# 修复文档（问题追踪与回归验证）

## 1. 背景
- **项目**：frag-it
- **模块**：历史记录（localStorage）/ 使用日历（/calendar）

## 2. 问题现象
- **现象 A（历史记录易被“清空”）**：在迭代/修复某个 bug 后，用户的过往记录在页面中不再显示。
- **现象 B（难以验证修复有效性）**：当记录已经变为空后，即使修复了“防误清空”的逻辑，也无法通过现有数据直接判断修复是否生效。

## 3. 复现方式（常见触发）
- **触发 1：读取失败后被覆盖**
  - 应用启动时从 `localStorage` 读取历史。
  - 如果读取/解析/校验失败，读取结果可能变为 `[]`。
  - 随后某些实现会在初始化后立刻执行保存逻辑（例如 `useEffect` 里 `saveHistory(history)`），从而把 `[]` 覆盖写回 `localStorage`，造成数据“被清空”。
- **触发 2：结构升级导致校验失败**
  - 历史数据结构在版本升级后新增/变更字段。
  - 旧数据不满足新校验规则，读取后被过滤为空数组。
  - 如果随后写回，会导致用户旧数据丢失。

## 4. 根因分析
- **根因**：将“读取失败/数据不兼容”与“用户确实没有历史记录”混为一谈。
  - 读取失败时仍然执行持久化写回，会把潜在可恢复的数据覆盖掉。
  - 严格校验 + 自动写回属于高风险组合。

## 5. 修复策略（已实现）
- **策略 1：读写分离并返回状态**
  - 新增 `loadHistoryResult()`：返回 `{ items, ok, key }`。
  - `ok=false` 表示读取/解析/校验失败，业务侧不得自动写回覆盖。
- **策略 2：读取失败禁止持久化写回**
  - `useHistory` 中引入 `canPersistRef`：仅当初次读取 `ok=true` 时才允许后续 `saveHistory`。
- **策略 3：旧 key 兼容与迁移**
  - 增加 `LEGACY_HISTORY_KEYS`，允许从旧 key 回退读取。
  - 若读到旧 key 且新 key 为空，则自动迁移写入新 key（只做一次，且不覆盖已有新数据）。

## 6. 验证方法（用于回归测试）
- **方法 A：模拟存在历史后刷新/重载**
  - 先在页面内粉碎几次，确认 `/calendar` 可看到记录。
  - F5 刷新 `/calendar`，记录仍应存在。
  - 关闭浏览器标签页后重新打开，记录仍应存在。
- **方法 B：模拟“异常数据”不被覆盖**
  - 在浏览器 DevTools -> Application -> Local Storage 中找到历史 key。
  - 手工改成一个不合法/不兼容的 JSON。
  - 刷新页面，应用不应把该 key 自动覆盖写成 `[]`。

## 7. 已知限制与用户影响
- **限制**：如果在修复之前，用户浏览器的 `localStorage` 已经被覆盖成 `[]`，则旧数据无法从应用侧自动恢复。
- **建议**：后续可以考虑提供“导出/导入备份”能力，让用户在本地保留可恢复的数据副本。

## 8. 后续建议（可选）
- **建议 1：备份槽位**
  - 每次写入前，将旧值备份到 `frag-it.history.backup.v1`，以便回滚。
- **建议 2：结构版本化与迁移**
  - 在历史记录顶层引入版本号（例如 `{ version, items }`），升级时做显式迁移而不是直接过滤。
